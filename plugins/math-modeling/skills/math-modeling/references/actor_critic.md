# Actor-Critic 机制使用指南

## 概述

Actor-Critic 是贯穿本插件的核心自我优化机制。每个关键阶段（问题分析、高层建模、公式生成）都使用此机制来迭代改进输出质量。

## 工作流程

每个 Actor-Critic 循环包含 3 个步骤：

### 1. Actor（生成）

基于输入信息生成初步输出。要求：
- 全面、深入、有洞察力
- 不回避复杂性，充分展开论述
- 使用 LaTeX 公式表达数学内容
- **推荐使用结构化格式**：变量定义用表格、假设条件用编号列表、公式带编号
- 连贯段落用于论述和推理，但关键定义和公式应独立呈现以提高可读性
- 段落间保持逻辑连贯

**格式建议**：
- 假设条件：使用编号列表 (1)(2)(3)...
- 变量定义：使用表格或 $x_i$ 形式的明确标注
- 公式：带编号的 LaTeX 公式，如 $\text{(1)} \quad f(x) = ...$
- 论述推理：连贯段落

### 2. Critic（批评）

对 Actor 的输出进行严格批评，关注：
- **深度**：是否超越表面观察？假设和隐含约束是否充分考虑？
- **新颖性**：是否提供了新见解，还是复述已知方法？
- **严谨性**：逻辑是否一致？数学表述是否精确？
- **上下文意识**：是否考虑了实际约束和现实世界的影响？
- **数据意识**：是否充分利用了可用数据？数据处理是否合理？

批评时**必须给出具体的改进方向**（指出问题所在 + 建议改进的方向，但不提供完整方案）。例如：
- "对变量 $x$ 的边界条件讨论不充分，应考虑物理约束下的取值范围"
- "假设(3)忽略了时间延迟效应，建议引入时滞项分析"
- "公式(5)的量纲不一致，需检查左边和右边的单位"

#### 独立 Critic 模式（Stage 1 & Stage 2 使用）

在问题分析（Stage 1）和高层次建模（Stage 2）阶段，Critic 应使用 Agent 工具派生**独立 subagent** 执行，而非由同一 agent 内联扮演。

**原因**：同源自我批评容易继承 Actor 的盲点。独立 subagent 拥有不同上下文，能发现 Actor 忽略的问题。

**执行方式**：
```
Agent(subagent_type="general-purpose", prompt={
  "你是一名严格的数学建模评审专家（Critic 角色）。请对以下内容进行批评。",
  "",
  "## 审查标准",
  "- 深度：是否超越表面观察？",
  "- 新颖性：是否提供了新见解？",
  "- 严谨性：逻辑是否一致？数学表述是否精确？",
  "- 上下文意识：是否考虑了实际约束？",
  "- 数据意识：是否充分利用了可用数据？",
  "",
  "## 被审查内容",
  "{Actor 的完整输出}",
  "",
  "## 问题背景",
  "{问题的简要描述，1-2 段}",
  "",
  "## 输出要求",
  "1. 逐一指出具体问题（附位置引用）",
  "2. 每个问题给出改进方向（指出方向，不提供完整方案）",
  "3. 最后给出总体评价：是否存在重大问题需要追加一轮",
  "4. 直接输出批评内容，不要有多余的寒暄"
})
```

**约束**：
- subagent 的 prompt 中只包含 Actor 的最终输出，不包含 Actor 的推理过程或草稿
- Stage 3（任务求解）的 Critic 保持内联模式，因为任务可能并行执行，需要控制 subagent 总数

### 3. Improvement（改进）

根据 Critic 的批评改进输出：
- 直接给出改进后的完整版本
- 不重复提及之前版本的缺陷
- 改进后的版本应是一个独立完整的文档

## 轮次配置

| 阶段 | 默认 Actor-Critic 轮次 |
|------|----------------------|
| 问题分析 | 2 轮 |
| 高层建模 | 2 轮 |
| 任务公式生成 | 1 轮 |

### 自适应轮次规则

默认按上表执行。但如果最后一轮的 Critic **仍发现重大问题**（如：核心逻辑错误、遗漏关键假设、公式推导错误），则追加 1 轮 Critic → Improvement，最多追加 1 轮（即最多 3 轮）。

判断"重大问题"的标准：
- 数学公式存在推导错误或量纲不一致
- 遗漏了问题明确要求的约束或条件
- 核心假设与问题背景明显矛盾
- **不**包含表述不够优美、分析可以更深入等主观判断
